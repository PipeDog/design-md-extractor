# DESIGN.md Extractor Chrome 插件设计稿

## 1. 目标

构建一个 Chrome 插件，针对用户当前打开的网页执行 UI 分析，并在浏览器中直接导出下载 `DESIGN.md`。

插件必须支持两种分析模式：

- 本地分析模式：用户未配置模型参数时自动启用，仅依赖本地规则分析。
- 增强分析模式：用户已配置模型参数时自动启用，基于本地规则结果再调用远程模型增强输出。

用户只关注结果，不需要理解底层技术实现。界面只需清楚表达当前处于「基础分析」还是「增强分析」。

## 2. 产品边界

### 2.1 V1 范围

- 分析当前活动标签页的单页面视图。
- 识别页面类型、信息层级、视觉风格、常见组件模式与基础可用性信号。
- 自动生成 Markdown 文档并下载到本地。
- 支持本地规则模式与远程模型增强模式的自动切换。
- 支持在设置页配置模型参数、导出偏好与站点排除规则。

### 2.2 明确不做

- 不做整站爬取。
- 不做多页面合并分析。
- 不做账号体系、云同步、历史记录云端保存。
- 不做复杂截图视觉识别流水线。
- 不做非 Chrome 浏览器兼容适配。

## 3. 用户流程

### 3.1 首次使用

1. 用户安装插件并打开任意网页。
2. 点击插件图标，Popup 展示当前页面、分析模式和主操作按钮。
3. 若未配置模型参数，Popup 显示本地模式提示，并引导用户前往设置页配置 AI 增强能力。
4. 用户点击「分析并导出 DESIGN.md」。
5. 插件完成分析后直接下载 Markdown 文件到本地。

### 3.2 已配置 AI 的使用流程

1. 用户在设置页填入模型服务地址、协议、模型名和 API Key。
2. Popup 自动显示为增强分析模式。
3. 用户点击分析按钮后，插件先执行本地结构化提取，再调用远程模型生成更完整的设计文档。
4. 文档生成后直接下载到本地。

## 4. 页面与交互设计

### 4.1 Popup

Popup 是用户的主操作入口，采用三段式结构：

- 顶部状态卡：显示当前站点、当前模式、页面可分析状态。
- 中部操作区：唯一主按钮「分析并导出 DESIGN.md」，附带进度状态。
- 底部摘要区：显示最近一次分析的结果摘要、下载状态和设置入口。

关键交互规则：

- 页面存在但未配置 AI：显示轻提示，不阻止分析。
- 页面不可分析（如 `chrome://`、扩展页等）：禁用主按钮并展示原因。
- 分析中：按钮进入 loading 状态，显示阶段进度。
- 分析成功：自动下载文件，并展示简短摘要。
- 分析失败：显示明确错误与恢复路径。

### 4.2 设置页

设置页用于维护 AI 配置与偏好，不承担分析入口职责。结构如下：

- 配置总览卡
- 模型连接表单
- 分析与导出偏好
- 隐私与数据说明
- 站点规则

表单应强调两个状态：

- 已配置完整参数，可使用增强分析。
- 参数不完整，将退回本地规则分析。

## 5. 功能设计

### 5.1 页面采集

内容脚本只采集生成设计文档所需的最小必要信息：

- 页面标题、URL、语言、meta 信息
- 可见标题与主要文本块
- 关键布局容器与段落层级
- 导航、按钮、表单、卡片、列表、表格等常见组件线索
- 颜色、字体、间距、圆角、阴影等样式信号
- 图片、链接、交互元素数量与分布

采集原则：

- 优先可见区域与语义节点
- 限制节点数量和文本长度
- 跳过脚本、样式、隐藏节点和明显噪声内容
- 对潜在敏感输入进行脱敏

### 5.2 本地分析

本地规则引擎负责输出结构化分析结果，包括：

- 页面类型推断
- 视觉风格摘要
- 层级结构摘要
- 设计 token 提取
- 组件模式识别
- 基础可用性提示
- 生成基础版 `DESIGN.md`

本地分析结果既能直接导出，也能作为 AI 增强模式的输入。

### 5.3 AI 增强分析

当且仅当用户已配置完整模型参数时启用。

AI 层不直接消费原始 DOM，而是消费本地规则清洗后的结构化快照，以降低噪声和成本。远程模型负责：

- 提炼页面定位与设计意图
- 整合视觉语言和交互模式
- 输出更自然、更完整的设计文档语义层
- 在保留客观提取结果的基础上给出建议

### 5.4 Markdown 导出

插件分析完成后直接在浏览器中触发下载，文件名默认格式为：

`{domain}-DESIGN.md`

文档统一包含以下章节：

1. 页面概览
2. 设计摘要
3. Visual Tokens
4. Component Patterns
5. Layout & Hierarchy
6. Interaction Notes
7. Accessibility & UX Notes
8. Design Recommendations
9. Appendix

## 7. 协议支持

V1 同时支持两类协议：

- `Chat Completions`
- `Responses API`

内部实现采用统一 Provider 抽象层，将不同协议的请求构造和响应解析收敛到统一的内部结果对象，避免在业务层传播协议差异。

## 8. 架构设计

### 8.1 模块划分

- `popup`
  - 状态展示、启动分析、显示摘要
- `options`
  - 管理 AI 配置和偏好
- `content`
  - 页面信息采集
- `background`
  - 分析调度、远程请求、下载输出
- `shared/core`
  - 本地分析、Markdown 生成、协议适配、配置校验

### 8.2 数据流

1. Popup 请求后台启动分析。
2. Background 读取设置并判定模式。
3. Background 向 Content Script 请求结构化页面快照。
4. Background 调用本地分析器生成基础结果。
5. 若 AI 配置完整，则调用 Provider 适配器请求远程模型，合并增强结果。
6. Background 生成最终 Markdown。
7. Background 触发浏览器下载，并把摘要状态回传 Popup。

## 9. 错误处理

必须覆盖以下失败场景：

- 当前页面不支持采集
- 内容脚本注入失败
- 页面快照为空或噪声过高
- AI 配置缺失或非法
- 远程接口超时 / 认证失败 / 返回格式错误
- Markdown 生成失败
- 下载失败

错误提示原则：

- 提示原因必须用户可理解。
- 提供下一步动作，例如「前往设置页补全配置」。
- AI 失败时优先回退到本地输出，而不是直接中断。

## 10. 测试策略

V1 采用以核心逻辑为主的测试策略：

- 配置校验测试
- 页面快照摘要测试
- 本地分析结果测试
- Markdown 生成测试
- 双协议请求构造与响应解析测试

扩展壳层不做重量级端到端自动化，优先保证共享核心模块可测试、可验证。

## 11. 可行性结论

该方案可行，且适合在当前仓库中以 Chrome Manifest V3 扩展的形式落地。

关键前提如下：

- 采用单页分析而非整站抓取，控制采集复杂度。
- 采用本地规则优先、AI 增强可选的混合架构。
- 通过统一协议适配层同时支持 `chat_completions` 和 `responses`。
- 将页面采集、分析逻辑与 UI 壳层解耦，以降低实现和测试风险。
